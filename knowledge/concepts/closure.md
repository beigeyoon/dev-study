## 한 줄 정의
함수가 *태어난 곳*의 변수를 기억하고 그 변수를 살려둔 채 들고 다니는 것. 함수의 변수 접근은 *어디서 호출되는가*가 아니라 *어디서 정의(탄생)됐는가*로 결정된다(렉시컬 스코프). 클로저가 잡은 변수는 콜 스택 프레임이 pop돼도 **힙**에 남아 살아남는다.

## 깊은 설명
- **출발한 모순:** `setup(){ let message='안녕'; setTimeout(()=>console.log(message),1000) }` — `setup()`은 즉시 끝나 스택 프레임이 pop되는데, 1초 뒤 콜백은 `message`를 멀쩡히 읽는다. "함수 끝나면 지역 변수 사라진다"는 *기본값일 뿐* 절대 법칙이 아니라는 신호.
- **왜 안 죽나 = 도달 가능성:** 안쪽 콜백이 `message`를 참조하고, 그 콜백이 바깥 함수보다 오래 산다(큐로 넘어가 나중 실행). 콜백이 살아있는 한 `message`도 버릴 수 없다. **변수 수명 = 프레임 수명이 아니라 "누가 아직 참조하나".**
- **어디에 사나 = 힙:** 클로저가 잡은 변수 묶음(렉시컬 환경)은 **콜 스택이 아니라 힙**에 둔다. 스택 프레임은 `return` 시 pop되지만 힙의 묶음은 GC가 "도달 불가"로 판정할 때까지 산다. 큐에 있는 콜백은 그 힙 묶음을 **가리키는 참조**를 들고 있을 뿐(큐는 할 일 대기열이지 변수 저장소가 아니다).
- **스택 vs 힙:** 스택 = 짧게 사는 프레임(LIFO 자동 정리). 힙 = 오래 살아야 하는 것(GC가 도달 불가 시 회수). 클로저는 "지역 변수를 스택에서 힙으로 대피시키는" 장치로 볼 수 있다.
- **상태 유지(비동기 전용 아님):** `makeCounter(){ let count=0; return ()=>++count }` → 반환 함수가 `count`를 잡아 힙에 유지 → 호출마다 1,2,3. **하나의 `count`를 공유하는 살아있는 참조**(스냅샷 복사가 아니다). 함수가 사적이고 지속되는 상태를 갖게 하는 메커니즘.

## 내 파인만 설명 (직접 작성)
_(2026-06-21 세션, 콜 스택 미해결 질문에서 추론으로 도출)_

- 모순 앞에서 **"(a) 함수 끝나면 지역 변수 사라진다가 틀렸다"**를 직감으로 골라냄 → 한 칸 정밀화: (a)는 *보통은 맞고*, 이 케이스는 예외. 예외의 원인 = `setup` 안에서 태어난 콜백이 `message`를 참조 + 콜백이 더 오래 삶.
- "스택 프레임은 pop됐는데 `message`는 어디에?" → 큐로 한 번 빗나갔다가(큐=할 일 대기열이지 변수 저장소 아님) **힙**으로 교정. 콜 스택의 단짝.
- **응용 전이 성공:** `makeCounter`에서 "makeCounter는 곧장 pop돼도 반환 함수가 `count`를 참조하므로 `count`는 힙에 남아 계속 참조된다 → 1,2,3"을 자력 설명. 비동기(setTimeout) 사례에서 동기(상태 유지) 사례로 **같은 원리를 옮김.**
- **클로저의 형태(질문에서 도출):** 클로저는 코드로 쥐는 독립 객체/값이 *아니다* — `console.log(c)`는 *함수*만 찍히고(`typeof c === "function"`), 잡힌 변수는 안 보인다. 함수의 숨은 내부 슬롯 `[[Environment]]`가 힙의 환경 레코드를 가리킬 뿐. 코드로는 못 읽고(=사적, 이게 캡슐화의 핵심), Chrome DevTools `[[Scopes]] > Closure`로만 엿본다. → **"특정 가능한 건 함수, 배낭은 실재하지만 사적."**
- **배낭은 함수가 아니라 스코프(호출) 소유:** 한 호출 안에서 태어난 함수들은 *같은* 환경을 공유(`inc`/`dec`가 한 `count`를 나눠 1,2,1). 호출마다 새 환경 → 독립.
- **var/let 함정 자력 해결:** `for(var)` + setTimeout 3개 → `3,3,3`. ① 반복문(동기)이 끝까지 돌아 공유 `i`가 3으로 굳은 뒤 ② 콜백이 그 하나의 `i`를 늦게 읽음(스냅샷 ❌). `for(let)` → `0,1,2`: 반복마다 *새 바인딩*이라 콜백마다 자기 `i`(0/1/2)에 박제. **두 버전 다 클로저** — 차이는 "배낭 1개 공유 vs 3개 분리".

### 2026-06-22 복습 (첫 간격, mastery 3→4) — "배낭은 누구 소유인가" 새 앵글
앵커: `makeBank(balance){ return { deposit:n=>balance+=n, check:()=>balance } }` → `a=makeBank(100); b=makeBank(0); a.deposit(50); log(a.check(), b.check())` → **150, 0**.
- **무힌트 재구성:** ① 한 호출 안의 `deposit`·check`는 *같은* 배낭(같은 `balance` 참조) 공유 → deposit이 누른 걸 check가 봄. ② `b`는 `makeBank`의 *다른 호출* → 새 배낭 → 안 건드려짐. **배낭 = 함수 아닌 호출(스코프) 소유** 재확인.
- **밑단까지 팜(1차 원리):** `makeBank` 프레임이 `return` 직후 pop돼도 `balance`가 안 죽는 이유 = **deposit/check가 아직 참조 → 도달 가능 → 수명은 프레임이 아니라 참조**. 거처 = **힙**(스택 아님). 전이: `a=null`로 참조 끊으면 → 도달 불가 → **GC가 힙에서도 회수**. 클로저 트레이드오프(상태 캡처 편의 ↔ 안 끊으면 메모리 누수)의 뒷면을 자력으로 짚음.

## 연결 / 철학적 질문
- **resolves:** [콜 스택](call-stack.md)의 미해결 질문("프레임 사라진 뒤 콜백이 변수에 접근하는 법") — 답: 잡힌 변수는 힙으로 대피해 프레임과 따로 산다.
- **builds-on:** 콜 스택 / 힙 메모리 모델. **대조:** 스택(자동 pop) ↔ 힙(GC). **연결:** [async/await](async-await.md)·[매크로태스크 큐](macrotask-queue.md)의 콜백이 변수를 잡는 것도 전부 클로저.
- **why 중요:** 모듈 패턴(사적 변수), React `useState`/이벤트 핸들러가 렌더 시점 값을 기억, 디바운스/스로틀, `once`/메모이제이션 — 전부 클로저.
- **트레이드오프:** 안 풀리는 참조 = 메모리 누수 위험(콜백/리스너 안 떼면 힙이 안 비워짐). 편의(상태 캡처)와 메모리의 교환.
