## 한 줄 정의
HTTP가 **stateless**(요청마다 독립, 서버가 직전 요청을 잊음)인데도 "로그인 상태가 유지되는 것처럼" 보이게 만드는 장치. 실은 유지가 아니라 **매 요청마다 신분증(티켓)을 자동 재제출**하는 것. 그 티켓을 *어디에 담느냐* = **쿠키**, 티켓 안에 *무엇이 들었느냐* = **세션 vs 토큰**.

## 깊은 설명

### 출발 문제: stateless라 매번 잊는다
[HTTP](http.md)는 요청마다 독립이다. 로그인 응답을 보낸 순간 서버는 너를 깡그리 잊는다. 그래서 다음 요청 `GET /mypage`만 보면 서버 입장에선 *생판 처음 보는 사람*과 구별이 안 된다. → **두 번째 요청이 "나 아까 그 사람"임을 스스로 증명해야 한다.**

### 쿠키 = 티켓의 운반 수단 (브라우저가 자동)
서버가 상태를 기억 안 한다는 전제에서, **클라이언트가 증명을 들고 다녀야** 한다. 그 운반체가 쿠키이고, [HTTP](http.md)에서 배운 **헤더 칸**에 그대로 얹힌다(`Host` 헤더와 같은 자리).

```
로그인:   클라 ──POST /login (id+pw)──►  서버
         클라 ◄──200 + Set-Cookie:티켓── 서버   ("이거 보관해둬")
이후매번:  클라 ──GET /mypage + Cookie:티켓──► 서버  (브라우저가 자동으로 붙임)
```
- **출처:** 서버가 로그인 응답 헤더에 `Set-Cookie: <티켓>`. = "받아서 저장해."
- **자동:** 한번 저장된 쿠키는 그 도메인 가는 **모든 요청에 브라우저가 알아서** `Cookie: <티켓>`로 붙인다. 개발자가 매번 챙길 필요 없음.
- 그래서 "로그인 유지"는 착시 — 매 요청마다 티켓을 자동 재제출하는 것. (이벤트 루프가 "동시성을 흉내" 낸 것과 같은 결: stateless는 그대로인데 *상태가 있는 것처럼* 흉내.)

### 티켓 안에 뭐가 드나 = 세션 vs 토큰
서버가 `Cookie: abc123`을 돌려받았을 때 abc123이 누구인지 아는 방식이 두 가지:

| | **세션 (가)** | **토큰 (나, 예: JWT)** |
|---|---|---|
| `abc123`의 정체 | 의미 없는 **번호표** | 정보 자체가 담긴 **봉인된 쪽지** ("유니, 만료 14:00") |
| 상태(누가 로그인했나)를 누가 기억? | **서버**(수첩=메모리/DB) — 요청마다 뒤져 찾음 | **아무도 안 기억** — 토큰 까보면 끝 |
| 서버는? | **stateful** | **stateless 유지** (HTTP 철학을 서버까지 일관) |
| 강점 | **즉시 무효화** 쉬움 (수첩 줄 삭제) | **확장** 쉬움 (서버 여러 대여도 아무나 검증) |
| 약점 | 서버 메모리 부담·여러 서버 공유 곤란 | **즉시 무효화 어려움** (지울 대상이 없음) |

### 핵심 트레이드오프: 둘 다 깔끔히는 못 가진다
- **토큰의 치명적 약점:** 토큰이 털렸을 때 강제 로그아웃하려 해도, 서버가 아무것도 기억 안 하니 **지울 게 없다.** 봉인 쪽지는 만료시각까지 어디서든 유효 → 서버는 그게 "털린 것"인지 모른다. = stateless의 대가.
- **현실의 타협:**
  1. **만료를 짧게**(예: 15분) → 털려도 잠깐만 유효. 불편 보완으로 수명 긴 **refresh token**으로 access token 재발급.
  2. **무효화 목록(blocklist)을 서버에 둠** → 근데 이러면 서버가 다시 상태를 갖게 됨.
- → 무효화 능력을 되찾으려면 **stateless를 일부 포기**해야 한다. **확장성 ↔ 즉시 무효화**는 동시에 깔끔히 못 가진다. 그래서 세션이 안 죽고 둘 다 살아남았다.

### 언제 뭘 쓰나
- **세션:** 서버가 HTML 렌더링하는 전통 웹앱, 보안·즉시 무효화 중요(은행·관리자). 프레임워크 기본값(`PHPSESSID`·`JSESSIONID`·Django·Rails).
- **토큰:** 모바일/SPA→API(`Authorization: Bearer`), 마이크로서비스(공용 수첩 없이 각자 검증), 서드파티/OAuth("구글로 로그인").
- **둘 다(큰 실서비스 대부분):** 짧은 access token(stateless) + 서버 추적 refresh token(무효화) 하이브리드. 또는 웹은 세션 쿠키 / 모바일·공개 API는 토큰.
- **휴리스틱:** 단일 웹앱+즉시 무효화 → 세션 / 여러 서버·모바일·서드파티 → 토큰 / 큰 규모 → 하이브리드.

## 내 파인만 설명 (직접 작성)
_(2026-06-23 첫 접촉, 유도 추론 — HTTP stateless 떡밥에서 출발)_
- "두 번째 요청이 나를 어떻게 증명?" → **클라이언트가 헤더에 신분증을 얹어 보낸다 = 쿠키** 를 자력 도출. (오늘 배운 HTTP 헤더 3층을 바로 활용.)
- 그 티켓의 출처 = **로그인 응답(`Set-Cookie`)**, 운반 = **브라우저 자동(`Cookie`)** 으로 추론.
- **세션 vs 토큰을 자력으로 압축:** "세션 = 서버가 상태를 하나하나 기억(stateful) ↔ 토큰 = 정보가 토큰 자체에 담겨 서버가 안 기억해도 됨(stateless 가능)." → "누가 상태를 기억하는가" 축을 스스로 잡음.
- **약점은 막힘(학습 신호):** 토큰 무효화는 못 풀었고, "지울 게 없다 = 깔끔히 무효화 불가"가 정답임을 확인 → 트레이드오프(확장 ↔ 무효화)로 정리.
- _(주의: 첫 접촉 = 유도된 학습. 세션/토큰 갈림과 쿠키 메커니즘은 코치 시나리오로 유도. 다음 복습에서 무힌트로 stateless→쿠키→세션vs토큰→트레이드오프를 세우는지 점검. 압축에서 토큰의 대가(무효화)를 빠뜨린 비대칭을 다음에 메우는지도.)_

**복습/자력 재구성 (2026-06-24, 첫 간격):** stateless→증명 필요→**쿠키(헤더에 티켓)**→**브라우저 자동 재제출**→**세션 vs 토큰("누가 기억하나" 축)**→**트레이드오프**까지 골격을 무힌트로 세움(장치 이름 "쿠키"와 `Set-Cookie`/`Cookie` 메커니즘만 가볍게 보강). **지난번 비대칭 해소:** 6-23엔 빠뜨렸던 **토큰의 무효화 약점**("지울 게 없어 탈취돼도 못 죽임")을 이번엔 스스로 도출. 단 거꾸로 **토큰의 강점(확장성)**을 빠뜨려 한 번 유도("서버 여러 대일 때 뭐가 편한가")→"아무 서버나 까서 검증 가능, 공유 수첩 불필요"로 자력 회수. → 트레이드오프 양 날(**확장성 ↔ 즉시 무효화, 둘 다 깔끔히는 못 가짐**) 완성. learning→understood, m2→3, feynman 통과.

## 연결 / 철학적 질문
- **builds-on:** [HTTP](http.md) — HTTP가 stateless라는 **바로 그 성질**이 이 개념을 낳음. 해결책의 운반 수단도 HTTP 헤더(`Set-Cookie`/`Cookie`). HTTP의 트레이드오프 떡밥의 직접 후속.
- **why(설계 의도):** 서버를 stateless로 두면(토큰) 서버를 자유롭게 늘릴 수 있어 확장이 쉽다 — HTTP가 stateless로 보편성을 얻은 것과 같은 동기.
- **트레이드오프:** 확장성(stateless) ↔ 즉시 무효화(stateful)는 동시에 깔끔히 못 가진다. 하이브리드는 그 사이의 절충.
- **전이(transfer):** "상태를 어디에 둘까(서버 vs 들고 다니기)"는 인증 너머의 일반 설계 축 — 캐시, 분산 시스템에서도 반복.
