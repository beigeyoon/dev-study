## 진행한 것
- **Promise.all 병렬 vs 순차 (자력 재구성, 체감상 첫 세션).** "시계 찍기"(각 fetch가 *몇 초에 호출되나*) 렌즈로:
  - 버전 C(분리 호출 후 await) = **1초** 자력 도출(처음 3초 오답 → A는 await가 다음 줄 진입을 막아 fetchB가 t=1에 호출, C는 세 줄 사이 await 없어 셋 다 t=0 점화 → await pB 시 이미 끝남).
  - 빈칸 자가 충전: **"병렬성은 Promise.all이 아니라 await보다 먼저 다 호출해 in-flight로 만든 데서 나온다."** Promise.all=편한 집계기. 범인은 단어가 아니라 *호출 시점*.
  - **CPU 버전 = 3초**를 암기 아닌 원리로(단일 스레드 점유). 1차 원리 자기 말: **"네트워크 기다림은 CPU가 아니라 Web API가 처리, CPU는 완료 소식만 듣는다"** → 단일 스레드인데도 I/O는 겹치고 CPU는 못 겹침. 겹침은 실행이 아니라 기다림에서. 전이: 진짜 계산 병렬 = Web Worker. + fail-fast(all) vs allSettled 학습.
- **async/await 회귀 점검.** `1 / setTimeout(2) / async{3;await null;4} / 5` → `1 3 5 4 2` 정확 + `4`(await 뒤)가 `2`보다 먼저인 이유를 "마이크로큐 전부 비운 뒤 매크로 1개"로 즉답. ✅ **두 번 무너졌던 자리, 이번엔 안 무너짐.**
- **🆕 클로저 신규 학습 (콜 스택 미해결 질문에서 출발 → ingest + 파인만 통과).**
  - 모순(setup 프레임 pop됐는데 1초 뒤 콜백이 message 읽음) → "(a) 함수 끝나면 지역 변수 사라진다"가 *기본값일 뿐*임을 직감 포착 → 콜백이 message 참조 + 더 오래 삶 → message는 못 버림. **변수 수명 = 프레임이 아니라 도달 가능성.**
  - "스택 프레임 pop됐는데 message는 어디에?" → 큐로 한 번 빗나갔다(큐=할 일 대기열) → **힙**으로 교정.
  - 전이: makeCounter(동기 상태유지) 자력 설명. inc/dec 두 함수가 한 count 공유(1,2,1) → **배낭은 스코프(호출) 소유**.
  - **var/let 끝판왕** 자력 해결: var→`3,3,3`(배낭 1개 제자리 수정 + 콜백이 끝난 뒤 늦게 읽음), let→`0,1,2`(반복마다 새 바인딩). **두 버전 다 클로저**, 차이는 배낭 개수.
  - 추가 질문 2개(예리함): "클로저는 객체인가/찍을 수 있나" → 독립 객체 아님, 함수의 숨은 `[[Environment]]` 슬롯 + DevTools `[[Scopes]]`로만 엿봄. "let 각 배낭이 0/1/2로 초기화되나" → 맞음(새 바인딩, 직전 값 물려받아).

## 막힌 지점 / 보강한 것
- Promise.all: 버전 C를 처음 3초로 오판(= 병렬성을 `Promise.all` 단어 탓으로 귀속). "시계 찍기"로 호출 시점이 진범임을 스스로 교정.
- 클로저 var 함정: 처음 `0,1,2`로 예측(콜백이 예약 시점 값을 스냅샷한다고 오해) + 중간에 `2,3,4`(콜백마다 다른 값). 타임라인 추적(반복문 끝 → i=3 고정 → 동시에 읽음)으로 `3,3,3` 자력 도달.
- 미세 교정: let "i가 1씩 증가" 표현 → "반복마다 별개 i"로 정정(증가는 var의 모델).
