## 진행한 것 (전부 추론 기반 파인만)
- **콜 스택:** 스택 프레임 정체(지역변수·매개변수·return address), 중첩→LIFO의 *이유*, 재귀=스택 오버플로(무한루프와 구분), "콜 스택이 빈다"의 정확한 의미.
- **형제 vs 중첩 호출:** 어제 "안 쌓인다" vs 오늘 "쌓인다" 모순을 사용자가 직접 포착 → 형제/중첩 구분으로 해소.
- **큐 정체:** 클릭 콜백 = 매크로큐(별도 큐 아님). setTimeout/Promise.then 호출 자체는 동기(콜 스택), 콜백만 비동기로 큐 경유.
- **task 단위:** 스크립트 전체가 하나의 task → 줄 사이 빈 틈엔 이벤트 루프 안 끼어듦. ("0ms"의 진짜 의미 사용자가 정확히 추론.)
- **Web API(핵심 보강):** "기다림은 누가?" → JS 엔진 밖 브라우저 스레드(Web API/libuv)가 떠맡음. 다섯 등장인물(콜스택·Web API·매크로·마이크로·이벤트루프)로 싱글스레드 동시성 흉내를 재구성 성공.

## 막힌 지점 / 보강한 것
- 스택 프레임 개념 자체 모름 → 강의 후 재귀·블로킹은 추론으로 재구성.
- "스택 비면 바로 실행하면 되잖아?" 오해 → task 원자성으로 교정 (오늘의 best 질문).
- "기다림의 주체" 몰랐음 → Web API 보강이 최종 퍼즐.
