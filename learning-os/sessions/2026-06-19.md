## 진행한 것
- **Promise 신규 학습 (ingest + 파인만 통과, mastery 3).** 백지 암송 없이 앵커 추론으로 한 조각씩 재구성:
  - 상태 3종 pending→fulfilled/rejected, settled 불변.
  - **resolve=동사(행동), fulfilled=명사(상태)** 구분. resolve에 Promise 넘기면 "resolved≠fulfilled" 떡밥.
  - resolve/reject = 엔진이 인자로 건네주는 함수(executor 패턴). **값은 return이 아니라 인자로 전달.**
  - 버튼 누르는 주체 = 비동기 작업을 실제 수행하는 백그라운드 일꾼(Web API). fetch 내부의 숨은 executor.
  - `.then`은 등록만 → settled 순간 콜백이 마이크로큐로. `.then`은 새 Promise 반환(체이닝) + 자동 flatten.
  - **강점:** 상태·값을 박제 → 콜백이 타이밍 경쟁에서 자유로워짐(늦게 .then 붙여도 안 놓침).
- 기존 5개(async/await·microtask·event-loop·macrotask·call-stack·promise-all) 전부 양방향 연결.

### async/await 복습 (파인만)
- 앵커: `1/2/4/3/5` 출력 순서 + `await null`은 왜 멈추나.
- ⚠️ **회귀 발견:** "await 뒤 코드 = 매크로큐" 오해 재발(6-18 교정분). → `setTimeout` 대조로 모순 짚어 스스로 마이크로큐 재도출. → **review_due 1일 리셋(6-21), mastery 3 유지.**
- `await null`이 멈추는 이유를 Promise 상태기계로 끝까지 재구성 성공: `Promise.resolve(null)`=태어날 때부터 fulfilled → `.then` 등록만 → 콜백 무조건 마이크로큐 → "통과 없음".
- 놓친 것: 출력에서 `5` 누락(마지막 `.then`). 단 `3`→`5` 큐 등록순 논리는 정확.

### Promise 토대 재다지기 (요청: "여러 번 주고받아야 이해됨")
- 모호점 2개 자가신고 → 한 칸씩 소크라테스식으로 해소:
  - **왜 `Promise.resolve(null)`이 이미 fulfilled?** → "기다릴 일이 없어서"를 스스로 도출.
  - **`.then`은 값도 등록?** → 값은 상자(Promise)가 박제, `.then`은 콜백만 등록, settle 시 엔진이 박제값을 인자로 cb 호출.
- **상자 모델**(상태/값/할 일 3칸) 자기 말로 재구성 → promise.md에 박제. 인과 방향 교정: 값 채워짐→기다림 없음→fulfilled.
- 마무리 4칸 사슬(`await null`→상자→.then→마이크로큐→동기보다 뒤) 매끄럽게 자력 완주. ✅ 토대 굳음.
