## 진행한 것 (추론 기반 파인만)
- **앵커:** `1 / setTimeout(2) / async f{3, await Promise.resolve(), 4} / 5` 출력 순서.
- 초기 오해 3개 → 전부 추론으로 교정:
  1. "`f()` 호출 = 함수 전체가 나중으로 미뤄짐" → async 함수도 일반 함수처럼 콜 스택에서 **즉시 시작**, 첫 `await`에서만 멈춤.
  2. "`await` 뒤 코드 = 매크로큐" → desugar(`await x` ≈ `Promise.resolve(x).then(cb)`)로 `.then` 콜백 = **마이크로큐**임을 사용자가 직접 도출.
  3. "`await` 대상이 상수면 통과" → `.then`은 동기 실행 없음 → 상수·이미 끝난 Promise라도 **무조건 일시중단**.
- 응용 통과: `1 3 5 4 2`, 그리고 두 await 버전 `start A end B C`(await 2개 = 마이크로 2회 경유).

## 막힌 지점 / 보강한 것
- 핵심 교정점: async 함수의 **동기 머리 부분** 존재. 호출하면 첫 await까지는 평범하게 실행된다는 것.
- desugar로 "await = .then"을 잡자 마이크로큐 연결이 한 번에 풀림 → 어제 마이크로큐 페이지의 미해결 질문 해소.

# 세션 2 (같은 날, Promise.all 병렬 vs 순차 await)

reviewed: []
learned: [concepts/promise-all-vs-sequential.md]

## 오늘의 브리핑
- async/await 직후 자연스러운 다음 깊이로 사용자가 직접 선택.

## 진행한 것 (추론 기반 파인만)
- **앵커:** 각 1초짜리 fetchData 3개 — 순차 await vs `Promise.all` 총 시간.
- 초기 오해: "둘 다 3초, 차이 없다" → 교정 과정에서 핵심 질문들 폭발:
  - **best 질문:** "`await delay(1000)`인데 왜 delay가 먼저 호출돼?" → 평가 순서. `await`는 연산자, 피연산자(`delay(1000)`)를 **먼저** 평가해 Promise를 만들고 그 뒤 멈춘다. (`console.log(getName())` 비유로 사용자 납득.)
  - 그걸로 순차=3초(await가 다음 호출을 막음), 병렬=1초(배열 평가 시 세 호출 즉시 발사 → 타이머 동시 점화)를 사용자가 재구성.
  - **1차 원리 질문:** "Web API가 스레드 3개 돌려?" → 아니다. 타이머=장부 예약(≈0 스레드), 네트워크=OS non-blocking I/O. 동시성≠병렬성 구분 도입.
  - **경계 추론(스스로):** `await delay` 대신 1초 CPU 루프면 `Promise.all`도 3초 → 정확히 추론. Web API는 정해진 비동기 창구만 받음, 평범한 계산은 메인 스레드 독점 → Web Worker 필요로 전이.

## 막힌 지점 / 보강한 것
- 평가 순서(연산자 vs 피연산자) 개념 → 비유로 교정, 이게 전체 잠금 해제.
- "동시 대기 = 멀티스레드"라는 직관 → "기다림은 일이 아니라 예약"으로 1차 원리 교정.

## 결과
