## 진행한 것
- **Promise 복습 (1순위, 첫 직접 점검).** 앵커: `new Promise(res=>{log A; res(X); log B}); log C; p.then(log v); log D` 출력 순서.
  - **어려운 절반은 처음부터 견고:** `.then`=등록(실행 ❌)·resolve가 먼저 눌려도 박제값 기억해 마이크로큐行·콜스택 비운 뒤 `v=X` 실행.
  - **빈칸 = executor 동기 타이밍:** 처음 `C A B D X`로 오답(executor가 `.then` 시점/이후에 도는 줄 착각). 교정 추론: `fetch`가 즉시 요청을 쏘려면 executor가 이미 돌았어야 + `new Promise(fn)`=그냥 동기 함수 호출(`forEach(fn)`처럼 *지금*) → executor는 `new Promise` 평가 그 순간 통째 실행 → **A B C D X**.
  - **`.then` 2단계 재확인:** ① promise에 cb 등록 → ② settled되는 순간 마이크로큐 적재. 이 퍼즐은 이미 settled라 ①②가 거의 동시 → "바로 적재"처럼 보였을 뿐. = 늦게 붙은 `.then`도 안 놓치는 원리.
  - 결과: 통과, m3 유지(빈틈 실재 → 4는 아직), review_due 1→3일(6-25). 페이지에 교정 박제.
- **클로저 복습 (첫 간격, "배낭=누구 소유" 새 앵글).** 앵커: `makeBank(balance){return{deposit:n=>balance+=n, check:()=>balance}}` → `a=makeBank(100); b=makeBank(0); a.deposit(50)` → **150, 0**.
  - 무힌트: ① 한 호출 안 deposit·check는 같은 배낭(같은 balance) 공유 ② b는 다른 호출→새 배낭. **배낭=함수 아닌 호출(스코프) 소유.**
  - 1차 원리까지 팜: makeBank 프레임 pop돼도 balance 안 죽는 이유=참조됨→도달 가능→**수명=참조**, 거처=**힙**, `a=null`이면 도달 불가→**GC 회수**. 트레이드오프(상태 캡처 ↔ 메모리 누수)의 뒷면 자력 포착.
  - 결과: 통과, **m3→4 상향**(빈틈 없이 설명+트레이드오프/전이=§6 4 기준, Promise와 차등), review_due→6-25.
- **🆕 HTTP 신규 학습 (첫 non-frontend 영역, network 개방).**
  - **연결 먼저(§8):** fetch→pending Promise→Web API가 네트워크 떠맡음→resolve. 며칠간 본 사슬의 *바깥 칸*이 HTTP임을 다리로 제시.
  - **유도 추론으로 구조 도출:** "서버 입장에서 뭐가 필요한가" → method(동사) 자력 + path(명사, URL에 이미) → POST 시 데이터 어디? → **body** 자력 → 헤더(메타데이터)는 코치 유도. **요청/응답 둘 다 3층(시작줄/헤더/바디)**, 응답 첫 줄만 status code.
  - **status code:** "어느 정도 규격화" 직감 → 첫 자리 분류(2 성공/3 리다이렉트/4 클라 잘못/5 서버 잘못) 정리.
  - **다리 완성 + 함정:** fetch의 Promise는 404·500에도 **resolve**(응답 도착 자체가 네트워크 성공). reject는 응답이 아예 못 올 때만. → `if(!res.ok)` 필요. "resolve=완료 소식"이 HTTP에선 "답장 도착"으로 매핑.

## 막힌 지점 / 보강한 것
- Promise: executor가 `new Promise` 평가 순간 *동기* 실행됨을 한 번 놓침(C를 A·B 앞에 둠). fetch/함수호출 유추로 자력 교정. (페이지에 "생성 즉시 동기 실행"이라 직접 써뒀던 자리 = 약한 지점이었음, 이제 굳힘.)
- HTTP: 헤더의 필요성은 스스로 못 떠올려 코치가 유도(body 포맷·인증). 첫 접촉=유도된 학습 → status learning·m2·feynman 미통과로 정직하게 매김. 내일 첫 간격에서 무힌트로 3층 세우는지 점검.
