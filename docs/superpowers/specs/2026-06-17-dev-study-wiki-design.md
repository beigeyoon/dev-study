## 1. 비전 & 목표

karpathy의 "LLM 위키" 패턴(RAG처럼 매번 원문에서 재발견하지 않고, LLM이 점진적으로 쌓고 유지하는 **복리(compounding) 지식 아티팩트**)을 기반으로, 단순 지식 저장소를 넘어 **능동적 멘토/선생님** 역할을 하는 학습 시스템을 구축한다.

**최종 목적**: 대체 불가능한 풀스택 프로덕트 개발자로 성장. 프론트엔드를 넘어 네트워크/OS/AI/CS/백엔드/DevOps + 개발자 마인드까지 다채롭게 체득.

**3대 차별점** (단순 저장소와의 차이):
1. **능동 코치**: 지식 공백을 분석해 "다음엔 이걸 공부해"라고 먼저 제안
2. **체득 프로세스**: 파인만 기법(추론 기반) + 연결/철학적 질문으로 "이해의 착각"을 깨뜨림
3. **학습 루틴 체계화**: 데일리 세션 + 이벤트 기반 + 간격 반복 복습

## 3. 아키텍처 — 3레이어 + 마인드 트랙

지식(knowledge)과 학습 운영(learning-os)을 **물리적으로 분리**한다. 미래 웹앱이 둘을 각각 "지식맵"과 "성장 대시보드"로 시각화할 수 있게 하기 위함.

```
dev-study/
├── CLAUDE.md              # 헌법(Schema): 규칙·워크플로우·컨벤션. 매 세션 Claude가 읽음
├── index.md              # 모든 지식 페이지 카탈로그 (카테고리별 1줄 요약 + mastery 집계)
├── log.md                # append-only 연대기 (ingest/session/query 기록)
│
├── sources/              # 📥 RAW SOURCES (불변 — 읽기만, 수정 안 함)
│   ├── articles/         #   글/아티클/공식문서 원문 메모
│   ├── problems/         #   실무 중 마주친 문제 기록 (날것)
│   └── _inbox/           #   이동 중 던져둔 처리 전 캡처 (내용물)
│
├── knowledge/            # 🧠 KNOWLEDGE LAYER (karpathy substrate)
│   ├── concepts/         #   개념 페이지
│   ├── topics/           #   분야/영역 페이지 (network, os, ai …)
│   ├── entities/         #   도구·라이브러리·제품
│   └── synthesis/        #   분야를 가로지르는 통합 페이지
│
├── learning-os/          # 🎯 LEARNING OS LAYER (능동 코치)
│   ├── roadmap.md        #   풀스택 로드맵 (분야별 토픽 트리)
│   ├── gaps.md           #   코치가 식별한 지식 공백 + 우선순위
│   ├── backlog.md        #   학습 위시리스트 (배우고 싶은 주제 이름/의도)
│   ├── review-queue.md   #   복습 큐 (간격 반복)
│   ├── goals.md          #   큰 목표/방향성
│   └── sessions/         #   데일리 세션 로그 (YYYY-MM-DD.md)
│
└── mindset/              # 🧭 MINDSET TRACK (독립)
    ├── principles/       #   설계 철학·원칙 (체득한 것)
    └── reflections/      #   회고·의사결정 기록
```

**원칙:**
- `sources/`는 불변. Claude는 여기서 읽고 `knowledge/`에 가공물을 생성.
- 모든 지식 페이지는 frontmatter(메타데이터)를 가짐 → 숙련도·관계·상태가 기계 가독.
- `mastery`는 별도 파일이 아니라 각 페이지 frontmatter에 저장 → `index.md`가 집계 → 미래 웹앱이 그대로 소비.

## 4. 접근 도구 (운영 모델)

위키는 **git 저장소**로 만들어 **private GitHub**에 올린다. 이로써 데스크탑↔모바일 동기화, 버전 관리(학습 히스토리), 미래 도구의 토대를 확보.

```
하나의 private GitHub repo (= 위키)
├── 데스크탑:  Claude Code   ← 무거운 작업 (ingest, lint, 통합 정리)
└── 모바일:    Claude 앱      ← 데일리 세션, 캡처, 코칭 (GitHub 연결)
```

- **멘토(두뇌) = Claude.** 모바일 학습 = 모바일 Claude 앱이 repo에 접근해 대화.
- **이동 중 캡처** = `sources/_inbox/`(처리할 내용물) + `learning-os/backlog.md`(배우고 싶은 주제).
- Obsidian 등 별도 뷰어는 현재 범위 밖 (필요 시 같은 repo에 얹으면 됨). 포트폴리오용 웹앱이 장기적으로 뷰어/대시보드 역할을 대체.

## 5. 데이터 스키마 (frontmatter 규약)

미래 웹앱이 "공짜로 올라타는" 비결. 모든 지식 페이지 상단에 둔다.

```markdown
---
title: 이벤트 루프
type: concept                # concept | topic | entity | synthesis
domain: javascript           # frontend|backend|network|os|cs|ai|devops|...
knowledge_type: model        # model | reference
status: learning             # seed | learning | understood | mastered
mastery: 2                   # 0–5 (코치가 추론·재구성·응용 능력으로 평가·갱신)
importance: 4                # 1–5 (중요도 + 현업 활용도 → 복습 횟수·졸업선 조절)
review: auto                 # auto | retired | suspended (사람 우선 오버라이드)
feynman_passed: false        # 파인만(추론 기반) 통과 여부
created: 2026-06-17
updated: 2026-06-17
sources: [articles/jake-archibald-tasks.md]
related: [concepts/call-stack.md, concepts/microtask.md]
tags: [async, runtime]
review_due: 2026-06-20       # 복습 큐가 읽는 다음 복습일
---

## 한 줄 정의
## 깊은 설명
## 내 파인만 설명 (직접 작성)
## 연결 / 철학적 질문
## 미해결 질문
## 실전 사례 (실무 문제와 연결되면 추가)
```

### 5.1 필드 의미

- **`type`**: 페이지 종류. 웹앱이 분류·필터링에 사용.
- **`domain`**: 분야. 분야별 성장 그래프의 축.
- **`knowledge_type`**:
  - `reference` (참조형): API/문법/설정 등. **외울 필요 없음** — "존재 + 찾는 법"만. 파인만 대상 아님.
  - `model` (모델형): 동작 원리 등. **체득 대상** — 암기가 아니라 추론으로 재구성.
- **`status` 라이프사이클**: `seed`(씨앗) → `learning` → `understood` → `mastered`.
- **`mastery` 0–5**: 암송 정확도가 아니라 **추론·재구성·응용 능력** (§7.3 기준).
- **`importance` 1–5**: 중요도 + 현업 활용도. 복습 **횟수와 졸업선**을 조절 (§6).
- **`review`**: 사람 우선 오버라이드.
  - `auto`(기본): 알고리즘이 관리
  - `retired`: "이미 온전히 이해함 → 복습 큐 제외" (importance 무관)
  - `suspended`: 일시 정지 (나중에 `auto`로 복귀)
- **`related`**: 양방향 링크 → 지식 그래프 노드 연결.
- **`review_due`**: 다음 복습 예정일. 데일리 세션이 후보 선정에 사용.

## 6. 복습 알고리즘 (간격 반복 + 중요도)

**간격 진행은 모두 동일** (간격 효과 보존): `1 → 3 → 7 → 14 → 30 → 90일 …` 점점 벌어짐.

**`importance`는 "간격"이 아니라 "복습 횟수(졸업 시점)"를 조절:**

| importance | 목표 복습 횟수 | 졸업 후 |
|---|---|---|
| 5 (핵심·현업 다발) | 많이 (예: 8회+) | **졸업 안 함** — 90일마다 영구 유지 복습으로 계속 재등장 |
| 3 (보통) | 표준 (예: 4–5회) | 졸업 → 가끔(180일) 점검 |
| 1 (참고용) | 적게 (예: 2회) | 빠르게 졸업 → 복습 큐 은퇴 |

**작동:**
- 성공(파인만 통과)할 때마다 간격은 동일하게 다음 단계로 확장.
- 중요한 지식은 졸업선이 높아 더 많은 횟수를 거치고, 최고 중요도는 은퇴 없이 유지 복습 루프.
- 막히면 간격 리셋 (모두 동일).
- 데일리 세션은 `review_due` 지난 것 중 `importance` 높은 순으로 우선 노출.
- `review: retired`는 importance와 무관하게 큐에서 제외. `reference` 유형은 "존재를 아는가" 수준의 가벼운 점검만.

## 7. 워크플로우

각 워크플로우는 `CLAUDE.md`에 절차로 정의되어, 사용자의 한마디로 Claude가 수행한다.

### 7.1 Ingest (지식 흡수) — "이거 공부하고 싶어 / 이 문제 위키에 넣어줘"
1. 원문을 `sources/`에 보존 (`articles/` 또는 `problems/`).
2. 핵심을 **대화로 1차 학습** (조용히 저장만 ❌ — 되묻고 이해 확인하며 정리).
3. `knowledge/`에 개념 페이지 생성/갱신.
4. **[필수] 기존 위키와의 연결**: 기존 위키 전체를 스캔해 "전에 정리한 A·B와 이렇게 이어진다"를 먼저 제시하고 **양방향 `related` 링크**를 건 뒤 정리 마무리.
5. **연결 개념 제안**: 선행(prerequisites) / 이웃(siblings) / 대조(contrasts) / 상위·응용(builds-toward).
   - 위키에 없는 연결 개념은 `seed`로 추가하거나 `gaps.md`/`backlog.md`에 후보 등록.
6. `index.md` 갱신 + `log.md` 기록.
7. *실무 문제* 경로: `sources/problems/`에 날것 기록 → 근본 개념 추출 → 해당 개념 페이지에 "실전 사례"로 연결 → 그 개념 `importance` 자동 상향 검토.

### 7.2 데일리 세션 — "오늘 세션 시작" (매일의 진입점)
- 코치 브리핑 4갈래: ① 오늘 복습할 것(`review_due` × importance 순) ② `_inbox` 미처리분 ③ **백로그 환기** ("전에 X 공부하고 싶다 했지 — 오늘 할까?") ④ 추천 다음 학습(gaps 기반).
- 복습은 파인만 방식 (§7.x). 세션 끝에 `learning-os/sessions/YYYY-MM-DD.md` 기록 + `log.md` 한 줄.

### 7.3 캡처 triage — `_inbox` 처리
- 이동 중 던져둔 메모/링크를 ingest 대상으로 분류하거나 개념 페이지로 승격.

### 7.4 학습 백로그 — "이거 언젠가 공부하고 싶어"
- `backlog.md`에 주제 이름/의도를 0초 캡처 (선택: importance·메모).
- 데일리 세션이 능동 환기. 공부 시작하면 ingest로 승격.
- 코치도 공백 분석/연결 제안에서 나온 개념을 사용자 승인 하에 백로그에 등록.

### 7.5 Query (질문) — "X가 뭐야? / A랑 B 차이?"
- 위키 먼저 검색 → 인용과 함께 답변 → 가치 있는 탐구는 다시 위키 페이지로 적재.

### 7.6 코칭 / 공백 분석 — "내 약점 분석해줘" (데일리·위클리 자동)
- 전체 frontmatter 스캔 → 끊긴 연결, 방치된 `seed`, 로드맵 대비 빈 영역 식별 → `gaps.md` 갱신 → 다음 학습 제안.

### 7.7 Lint (위생 점검) — "위키 점검해줘" (가끔)
- 모순·구식 정보·고아 페이지·깨진 링크·`retired` 후보 점검.

## 8. 체득 메커니즘

### 8.1 파인만 기법 (추론 기반, 오픈북 허용)
- **백지 암송 ❌ → 단서 있는 추론 ⭕.** 코치가 앵커(상황·시작점)를 던지고, 사용자는 추론으로 풀어나감.
- **막히면 실패가 아니라 학습 신호** — 단계적 힌트. 막힌 지점이 진짜 이해 안 된 곳 → 거기만 보강.
- **오픈북 허용.** 평가 대상은 *외웠는가*가 아니라 **논리 사슬 재구성 / 왜인지 설명 / 응용**.
- `reference` 유형은 파인만 대상 아님 (model 유형만).
- 절차: 코치 요구 → 사용자 설명 → 코치가 빈틈 사냥(채점 ❌) → 막힌 부분 보강 → 통과 시 사용자 설명을 페이지에 박제(다음 복습 기준선), `feynman_passed`·`mastery`·`review_due` 갱신.

### 8.2 연결/철학적 질문 (계층적, 점점 깊어짐)
- **연결**: 전에 배운 X와 어떻게 이어지나, Y와 어떻게 다른가
- **설계 의도(why)**: 왜 이렇게 설계됐나, 다른 선택지는
- **트레이드오프**: 이걸 쓰면 무엇을 포기하나
- **1차 원리**: 이 개념이 풀려는 근본 문제는
- **전이(transfer)**: 전혀 다른 분야에 적용하면 (예: OS↔프론트엔드)
- 가치 있는 답 → 페이지 "연결/철학적 질문" 섹션에 보존. 못 한 질문 → "미해결 질문"에 남겨 다음 학습 떡밥.

### 8.3 mastery 평가 기준 (코치의 판단 규약)
- **0–1**: 읽기만 함 / seed
- **2**: 파인만 시도하나 빈틈 많음
- **3**: 빈틈 없이 설명 (understood)
- **4**: 연결·트레이드오프까지 설명, 응용 가능
- **5**: 1차 원리에서 재구성(암기 재현 ❌) + 타 분야 전이 (mastered)
